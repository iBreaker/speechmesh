# MeloTTS Local Bridge (First Pass)

This document describes the first-pass local bridge that exposes a WebSocket TTS facade backed by the existing MeloTTS HTTP server.

## Scope

- does not modify `speechmeshd` runtime code
- runs as a standalone helper process
- talks to `MeloTTS/scripts/local_tts_server.py` over HTTP
- exposes a simple provider-style WebSocket endpoint for SpeechMesh-side integration work

## Components

- upstream TTS server:
  - `http://127.0.0.1:7797/v1/tts`
  - `http://127.0.0.1:7797/healthz`
- bridge:
  - script: `scripts/melotts_ws_bridge.py`
  - websocket endpoint: `ws://127.0.0.1:8797/ws/tts`
  - health endpoint: `http://127.0.0.1:8797/healthz`

## Message Flow

1. client sends `hello`
2. bridge returns `hello.ok`
3. client sends `tts.start` with optional text
4. bridge returns `session.started`
5. client sends `tts.commit`
6. bridge calls Melo `/v1/tts`
7. bridge emits one or more `tts.audio.chunk` frames
8. bridge emits `session.ended`

Supported client message types:

- `hello`
- `tts.voices`
- `tts.start`
- `tts.input_text` or `tts.append`
- `tts.commit` or `tts.flush`
- `session.stop` or `tts.cancel`

Server message types:

- `hello.ok`
- `tts.voices.result`
- `session.started`
- `tts.input.accepted`
- `tts.audio.chunk` (base64 WAV bytes)
- `session.ended`
- `error`

## Run

```bash
cd /Users/breaker/src/speechmesh
source deploy/melotts-bridge.env.example
scripts/run_melotts_ws_bridge.sh
```

If you do not want to source env:

```bash
scripts/run_melotts_ws_bridge.sh \
  --python /Users/breaker/src/MeloTTS/.venv/bin/python \
  --listen 127.0.0.1:8797 \
  --upstream http://127.0.0.1:7797
```

## Smoke Test

```bash
scripts/test_melotts_ws_bridge.sh ws://127.0.0.1:8797/ws/tts
```

Expected output is a JSON object with:

- `ok: true`
- non-empty `sessionId`
- `chunkCount >= 1`
- non-trivial `audioBytes`

## Integration Notes

- This is a bridge/helper layer for current development; it is not yet wired into `speechmeshd` TTS routing.
- Audio is generated by Melo in one HTTP call and then chunked by the bridge into WebSocket events.
- `tts.audio.chunk` currently uses base64-encoded WAV bytes for compatibility and observability.
- The schema is intentionally minimal and can be adapted later to the final unified SpeechMesh TTS contract.
